# Architecture

**Analysis Date:** 2026-04-17

## Pattern Overview

**Overall:** Two-Agent ReAct (Reasoning + Acting) System with Streamlit UI

**Key Characteristics:**
- Dual-agent architecture: Persona Agent (real-time interaction) + Feedback Agent (post-session analysis)
- ReAct loop: Claude reasoning → tool calls → state mutations → loop until end_turn
- Tool-driven state management: agents modify interview state through tool execution, not direct function calls
- Langfuse v4 observability: @observe() decorators create nested spans for tracing each agent decision
- Graceful degradation: Supabase storage is optional; app functions without it

## Layers

**Presentation Layer (Streamlit UI):**
- Purpose: Four-step interview workflow with live Agent Brain debug panel
- Location: `app.py` (lines 21–310)
- Contains: Session state management, UI rendering (render_step_* functions), agent invocation
- Depends on: `agents/persona_agent.py`, `agents/feedback_agent.py`, `storage/session_store.py`
- Used by: PM user interacting with the trainer

**Agent Layer (ReAct Loops):**
- Purpose: Implement agent decision-making and tool orchestration
- Location: `agents/persona_agent.py`, `agents/feedback_agent.py`
- Contains: Message loop, tool call dispatch, stop condition handling (end_turn / tool_use / error)
- Depends on: `tools/persona_tools.py`, `tools/feedback_tools.py`, `prompts/persona_system.py`, `prompts/feedback_system.py`, Anthropic API
- Used by: Streamlit app to run interview turns and generate feedback

**Tool Layer (State Mutation):**
- Purpose: Execute agent-requested actions and modify interview state
- Location: `tools/persona_tools.py`, `tools/feedback_tools.py`
- Contains: Tool executors + state update logic (reveal_pain, update_resistance, compute_score, etc.)
- Depends on: Session state dict, persona data, metrics
- Used by: Agent layer via tool dispatch

**Prompt Layer (System Prompts):**
- Purpose: Context and instruction encoding for agents
- Location: `prompts/persona_system.py`, `prompts/feedback_system.py`
- Contains: Dynamic system prompts built from persona/metrics at runtime
- Depends on: Persona dict, metrics list
- Used by: Agents to guide Claude's behavior

**Storage Layer (Persistence):**
- Purpose: Cross-session learning — store sessions, load history for recurring pattern detection
- Location: `storage/session_store.py`
- Contains: Supabase client wrapper, save/load functions, aggregation (get_dashboard_stats)
- Depends on: Supabase SDK, environment credentials
- Used by: Feedback Agent (load_pm_history tool), Streamlit app (dashboard)

## Data Flow

**Interview Session Flow:**

1. **Setup (Context → Metrics Steps)**
   - PM enters context → `generate_persona()` creates hidden persona dict
   - `generate_metrics()` proposes 4 relevant metrics
   - PM selects metrics → creates `agent_session` dict (immutable across the interview)

2. **Interview Loop (One turn = one PM question)**
   - PM asks question → appended to `conversation_history` in `agent_session`
   - `run_persona_turn()` called with question + `agent_session`
   - ReAct loop: `_react_loop()` runs until Claude stops with "end_turn"
   - Each Claude iteration:
     - Tool calls trigger executors (e.g., `_reveal_pain()`) that mutate `agent_session` state
     - Tool results returned to Claude as observations
     - Claude re-reasons with updated state
   - Final response returned to UI; chat history updated
   - State changes (resistance, revealed_pains) visible in live stats bar

3. **Feedback Generation (Post-Interview)**
   - `run_feedback()` called with complete `agent_session` + conversation history
   - Feedback ReAct loop: tools call sequence defined in prompt:
     1. `load_pm_history()` → loads past sessions from Supabase
     2. `analyze_question_patterns()` (3 calls: depth, type_ratio, progression)
     3. `identify_missed_pains()`
     4. `match_pains_to_metrics()`
     5. `compute_score()` → sets `_final_score` in state
   - Final report generated by Claude
   - Session record saved to Supabase (if available)

4. **Progress Dashboard**
   - `get_dashboard_stats()` aggregates past sessions
   - Recurring missed pains detected via keyword matching

**State Management:**

- Session state is a plain dict passed by reference through all agent calls
- State mutations happen only in tool executors: `_reveal_pain()`, `_update_resistance()`, `_compute_score()`, etc.
- Langfuse observability decorators (@observe) capture inputs/outputs but don't affect state
- Streamlit session_state separate from agent_session (agent_session is nested dict within Streamlit's st.session_state)

## Key Abstractions

**Session Dict (agent_session):**
- Purpose: Represents the complete interview context — persona, metrics, progress, history
- Examples: `st.session_state.agent_session` (initialized in render_step_context, line 121)
- Pattern: Mutable dict passed by reference; agents read/modify it via tool execution
- Structure:
  ```python
  {
    "persona": {...},                       # Synthetic user (hidden)
    "metrics": [...],                       # Selected interview metrics
    "conversation_history": [...],          # Full message history (role, content)
    "resistance_level": int,                # Current persona openness (1-10)
    "revealed_pains": {pain_id: level},    # Tracked pain revelations
    "flagged_questions": [...],             # Questions marked good/bad for feedback
    "_pain_discovery_rate": int,            # Computed by feedback agent
    "_depth_score": int,                    # Computed by feedback agent
    "_metric_coverage_rate": int,           # Computed by feedback agent
    "_final_score": int                     # Computed by feedback agent
  }
  ```

**ReAct Loop (Persona & Feedback Agents):**
- Purpose: Iterative Claude reasoning with external tool integration
- Examples: `_react_loop()` in `agents/persona_agent.py` (lines 56–145), `run_feedback()` in `agents/feedback_agent.py` (lines 55–124)
- Pattern:
  1. Send messages to Claude with tools available
  2. If stop_reason == "end_turn": extract text, exit
  3. If stop_reason == "tool_use": extract tool calls, execute, append results, loop
  4. If other stop_reason: log error, exit
- Termination: max 10 iterations (persona), max 15 (feedback) to prevent runaway

**Tool Definition + Executor Pair:**
- Purpose: Claude sees JSON schema (definition), Python code executes (executor)
- Examples:
  - `PERSONA_TOOL_DEFINITIONS[0]` + `_reveal_pain()` in `tools/persona_tools.py`
  - `FEEDBACK_TOOL_DEFINITIONS[2]` + `_identify_missed_pains()` in `tools/feedback_tools.py`
- Pattern: Claude decides when/how to call; executor modifies session_state and returns string observation

**Langfuse Span Hierarchy:**
- Purpose: Observability tree showing which agent decisions led to which actions
- Examples: `@observe(name="persona-turn", as_type="agent")` creates a span; nested `@observe(name="react-loop", as_type="chain")` nests inside it
- Pattern: Each tool call becomes a child span via `get_client().start_as_current_observation(...)`
- Metadata: Token counts, resistance level, revealed pains appended via `update_current_span()`

## Entry Points

**render_step_context (app.py, lines 88–130):**
- Location: `app.py:88–130`
- Triggers: Initial page load; or "Recommencer" button at end
- Responsibilities:
  1. PM enters context text
  2. Calls `generate_persona()` → gets dict with role, pains, resistance
  3. Calls `generate_metrics()` → gets list of 4 metrics
  4. Initializes `agent_session` dict
  5. Transitions to "metrics" step

**render_step_interview (app.py, lines 165–242):**
- Location: `app.py:165–242`
- Triggers: After metrics selected; PM enters question via chat input
- Responsibilities:
  1. Displays live stats (resistance, pains discovered, questions asked)
  2. Renders chat history
  3. On new question: calls `run_persona_turn()` with question + `agent_session`
  4. Logs agent_log entries to brain_log for visualization
  5. At ≥3 questions: "End interview" button calls `run_feedback()`

**run_persona_turn (agents/persona_agent.py, lines 30–52):**
- Location: `agents/persona_agent.py:30–52`
- Triggers: Each PM question during interview
- Responsibilities:
  1. Builds system prompt from persona
  2. Appends PM question to conversation_history
  3. Calls `_react_loop()` to run one full agent decision cycle
  4. Updates Langfuse span with resistance_level + revealed_pains metadata
  5. Returns {"response": str, "agent_log": list}

**run_feedback (agents/feedback_agent.py, lines 25–135):**
- Location: `agents/feedback_agent.py:25–135`
- Triggers: PM clicks "End interview and get feedback"
- Responsibilities:
  1. Builds feedback system prompt from persona + metrics
  2. Triggers single message: "The interview session is complete..."
  3. Runs ReAct loop (up to 15 iterations) with feedback tools
  4. Tools sequence: load_pm_history → analyze_question_patterns (3x) → identify_missed_pains → match_pains_to_metrics → compute_score
  5. Returns {"report": str, "score": int, "agent_log": list}

## Error Handling

**Strategy:** Graceful degradation. Failures in Supabase, Langfuse, or API calls do not crash the app.

**Patterns:**

- **Supabase unavailable:** `_get_client()` returns None if credentials missing or SDK import fails (session_store.py:14–24). `save_session()` and `load_recent_sessions()` return False or [] respectively. Dashboard stats silently omitted.

- **LLM API errors:** Not explicitly caught; Anthropic SDK raises exceptions that would bubble up. Should be wrapped in try/except in production.

- **ReAct loop fallback:** If max iterations reached without end_turn, agent_log records error and exits with partial response.

- **Tool execution errors:** Returned as string observations to Claude (e.g., "Error: pain_id 5 does not exist"). Claude can re-reason and retry.

## Cross-Cutting Concerns

**Logging:** 
- Agent brain panel (`render_brain_log()` in app.py:49–76) visualizes agent_log events in real-time
- Each agent action type (llm_call, tool_call, observation, error) formatted for UI display
- Langfuse server-side traces capture full decision tree; local agent_log is UI-focused summary

**Validation:**
- Persona generation: JSON parsing with fallback `_parse_json()` handles code blocks
- Metrics: Claude returns JSON array; no schema validation (assumes Claude compliance)
- Session state: agent_session dict grows with computed fields (_final_score, _pain_discovery_rate); no pre-validation

**Authentication:**
- Anthropic API key: `ANTHROPIC_API_KEY` env var (required)
- Langfuse keys: `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, `LANGFUSE_HOST` (optional; @observe works without them)
- Supabase credentials: `SUPABASE_URL`, `SUPABASE_KEY` (optional; storage gracefully skipped)
